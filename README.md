# Simplified Trading Bot — Binance Futures Testnet (USDT-M)

A small, structured Python CLI application for placing MARKET, LIMIT, and
STOP orders on Binance USDT-M **Futures Testnet**, built with proper input
validation, logging, and error handling.

## Project Structure

```
trading_bot/
├── bot/
│   ├── __init__.py
│   ├── client.py            # Binance client wrapper (API layer)
│   ├── orders.py             # Order placement logic (business layer)
│   ├── validators.py         # CLI input validation
│   └── logging_config.py     # Centralized logging setup
├── cli.py                    # CLI entry point (command layer)
├── logs/                     # Log files land here (trading_bot.log)
├── requirements.txt
├── .env.example
└── README.md
```

The three-layer split (`cli.py` → `orders.py` → `client.py`) keeps the
Binance SDK isolated from business logic, and business logic isolated
from argument parsing — each piece is independently testable.

## Setup

### 1. Create a Binance Futures Testnet account
1. Go to https://testnet.binancefuture.com
2. Log in with GitHub (this is a testnet-only account, separate from a real Binance account).
3. Once logged in, generate an **API Key** and **API Secret** from the site (top-right, "API Key").
4. Use the "Faucet" on the testnet site to get free test USDT if your futures wallet balance is 0.

### 2. Install dependencies
```bash
python3 -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 3. Configure credentials
Copy `.env.example` to `.env` and fill in your testnet keys:
```bash
cp .env.example .env
```
```
BINANCE_API_KEY=your_testnet_api_key_here
BINANCE_API_SECRET=your_testnet_api_secret_here
```
Credentials can alternatively be passed directly via `--api-key` / `--api-secret`
flags, but `.env` is recommended so keys never end up in shell history.

## How to Run

### Place a MARKET order
```bash
python cli.py --symbol BTCUSDT --side BUY --type MARKET --quantity 0.01
```

### Place a LIMIT order
```bash
python cli.py --symbol ETHUSDT --side SELL --type LIMIT --quantity 0.05 --price 3200.5
```

### Place a STOP order (bonus order type)
```bash
python cli.py --symbol BTCUSDT --side BUY --type STOP --quantity 0.01 --price 60500 --stop-price 60400
```

### Skip the confirmation prompt (useful for scripting)
```bash
python cli.py --symbol BTCUSDT --side BUY --type MARKET --quantity 0.01 -y
```

### Dry run (validate & preview only, sends nothing to Binance)
```bash
python cli.py --symbol BTCUSDT --side BUY --type MARKET --quantity 0.01 --dry-run
```

By default (no `-y`), the CLI prints the order summary and asks for
confirmation before sending anything to Binance:

```
Order Request Summary:
  symbol    : BTCUSDT
  side      : BUY
  type      : MARKET
  quantity  : 0.01

Submit this order to Binance Futures Testnet? [y/N]:
```

### Sample successful output

## Logging

Every request, response, and error is logged to `logs/trading_bot.log`
(rotating, 1MB × 5 backups) in this format:

```
2026-07-01 17:48:09 | INFO  | trading_bot | REQUEST -> futures_create_order({...})
2026-07-01 17:48:09 | INFO  | trading_bot | RESPONSE <- futures_create_order: {...}
```

`logs/example_log_output.txt` in this repo shows the expected log format
(request → response → error entries) generated with a mocked Binance
client while developing this project offline. **Real log entries from
live testnet orders — one MARKET and one LIMIT — should be generated by
running the commands above with your own API credentials**, and should
replace/accompany that example file before final submission.

## Validation & Error Handling

- **Input validation** (`bot/validators.py`): symbol format, side
  (BUY/SELL), order type (MARKET/LIMIT/STOP), quantity > 0, price
  required for LIMIT/STOP, stop price required for STOP — all checked
  before any network call is made.
- **API errors**: `BinanceAPIException` / `BinanceOrderException` (e.g.
  invalid symbol, insufficient balance, invalid quantity precision) are
  caught, logged, and reported with a clear failure message.
- **Network errors**: connection failures/timeouts are caught separately
  so they're distinguishable from API-level rejections in the logs.
- **Unexpected errors**: caught as a last resort and logged with a full
  traceback (`logger.exception`) so nothing fails silently.

## Assumptions

- Binance Futures Testnet's spot/futures endpoints are reachable
  (`testnet=True` in `python-binance`, with `FUTURES_URL` explicitly
  pinned to `https://testnet.binancefuture.com/fapi` as a safeguard
  across library versions).
- Orders use `GTC` (Good-Til-Canceled) as the default `timeInForce` for
  LIMIT/STOP orders since none was specified in the requirements.
- Quantity/price precision (lot size, tick size) is assumed to be the
  caller's responsibility per symbol; Binance's API will reject values
  that violate a symbol's exchange filters, and that rejection is
  surfaced as a normal handled error rather than pre-validated client-side
  (fetching and caching `exchangeInfo` per symbol was out of scope for
  the ~60 minute estimate).
- STOP orders are implemented as `STOP` (stop-limit, requires both
  `price` and `stopPrice`) rather than `STOP_MARKET`, to demonstrate
  the more general case; `STOP_MARKET` support exists in the code path
  and can be selected by extending `--type` choices if preferred.

## Bonus Features Implemented

- **Extra order type**: STOP (stop-limit) orders in addition to MARKET/LIMIT.
- **Enhanced CLI UX**: colored output (via `colorama`, degrades
  gracefully if not installed), an interactive confirmation prompt
  before submitting, a `--dry-run` mode for safe validation/preview,
  and a `-y/--yes` flag to skip confirmation for scripted use.

## Running Tests / Verifying Without Spending Testnet Funds

Use `--dry-run` to validate the CLI layer and see the exact request
that would be sent, without contacting Binance at all:
```bash
python cli.py --symbol BTCUSDT --side BUY --type LIMIT --quantity 0.01 --price 60000 --dry-run
```
